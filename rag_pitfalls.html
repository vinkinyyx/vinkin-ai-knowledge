数据接入与解析

RAG 的第一道工序是 把非结构化文档（PDF、Word、扫描件）转化为机器可读的干净文本。这一步做不好，后续检索和生成都会崩。

⚠️ 坑：直接暴力提取 PDF 文本

现象：多栏排版文字混在一起、表格变成乱码、页眉页脚/水印污染正文。
原因：PDF 是视觉媒体，不是纯文本。传统 pdfminer 或 PyPDF2 按字符流读取，丢失了空间布局。
识别：检索时出现“左右栏拼接”的句子，或表格内容完全不可读。
解决：引入版面分析 + 智能 OCR。用轻量检测模型识别标题、正文、表格、页眉页脚，先过滤杂质。

✅ 最佳实践
  • 复杂表格：不要硬转文本，让大模型把表格转为 Markdown 或写一段摘要。
  • 图片/流程图：用多模态大模型（如 GPT-4o）生成文字描述，存入向量库。
  • 扫描件：先 OCR（如 PaddleOCR），再用版面分析还原阅读顺序。
  • 全局上下文注入：在每个切片开头拼上文档摘要，即使切碎也保留全文语义。
💡 工业级架构：解析层 + 结构还原层

第一层 解析层：多分流处理——原生 PDF 直接抓取文本，扫描件走 OCR，图表区域截图交给多模态模型做视觉转录。
第二层 结构还原层：把碎片重建为树状层级，统一转成 Markdown 格式，保留标题层级和表格行列对齐。

文档分块

分块（chunking）是 RAG 中最容易被低估的环节。切得太碎丢失语义，切得太粗超出模型窗口。

⚠️ 坑：固定 token 数暴力截断

现象：一句话被拦腰切断，表格被拆散，下一段开头的人称代词找不到指代对象。
原因：用 split_text(500) 这类死板方法，无视句子边界和文档结构。
识别：检索到的片段语义不完整，模型回答出现“张冠李戴”。
解决：采用 NLP 动态切分，识别句子边界、Markdown 标题、段落。

✅ 最佳实践
  • 重叠窗口：每个 chunk 之间重叠 10%~20%（约 100~200 token），像接力赛一样保证上下文连贯。仅此一步，准确率可提升 15%。
  • 父子文档分块：入库用短小的“子块”方便精确锁定，但喂给大模型时携带完整的“父文档”，上下文才够全。
  • 全局上下文注入：在每个切片开头拼上整个文档的摘要，即使切碎也带着全文语义。

向量化与向量数据库

将文本转为向量，并高效存储检索，是 RAG 的核心引擎。

⚠️ 坑：通用 Embedding 模型在垂直领域“失聪”

现象：医疗、法律、工业等专业术语检索不到，比如“胫骨平台骨折”被当成普通骨折。
原因：通用模型（如 text-embedding-ada-002）没学过领域语料。
识别：检索命中率低，尤其对稀有型号、专业黑话无效。
解决：两种方案——① 用业务数据微调 Embedding 模型（对比学习）；② 混合表示：同时存稠密向量 + 稀疏向量（类似 BM25），精准匹配关键词。

✅ 最佳实践
  • 量化压缩：生产环境用 int8 量化，内存省 4 倍。
  • 索引算法：首选 HNSW，像社交网络一样在海量向量中快速定位。
  • 增量更新：建立 MD5 哈希校验，实现 upsert 覆盖更新，避免脏数据堆积。

检索：关键词 + 向量 + 混合

纯向量检索无法满足生产需求，必须构建混合检索架构。

⚠️ 坑：向量分数与 BM25 分数直接相加

现象：BM25 得分是 10~20，向量得分是 0~1，直接相加导致向量分被淹没，混合检索名存实亡。
原因：不同检索器的分数量纲不同，缺乏统一归一化。
识别：混合结果几乎等同于纯关键词结果，语义匹配失效。
解决：用轻量级排序模型（LambdaMART）将多个特征映射到同一维度打分，而非拍脑袋定权重。

✅ 最佳实践
  • 前置过滤：按权限、时间等先踢掉不相关数据。
  • 多路召回：向量检索抓语义，BM25 抓精准术语，两路并行。
  • 策略路由：简单问题只走向量（毫秒级），复杂问题才开启重排序，好钢用在刀刃上。

重排

重排是消除“检索噪声”、保证最终输入质量的最后一道关卡。

⚠️ 坑：重排序模型太慢，系统并发崩溃

现象：每个查询重排耗时 200ms，并发 100 时直接超时。
原因：精排模型（如 cross-encoder）计算量大，不适合全量召回。
识别：接口延迟飙升，用户反馈“转圈”。
解决：策略路由——先粗排（向量+BM25）取 Top 30，再精排只算 Top 10。

✅ 最佳实践
  • 用更精密的模型（如 BGE-Reranker）对召回结果重新打分。
  • 结合 全局泛读 + 局部精读：Agent 先在大库里定位页码，再调用 ReadPage 工具精读该页。

答案生成

即使检索完美，大模型仍可能“一本正经地胡说八道”。

⚠️ 坑：模型忽略检索结果，自行编造

现象：回答内容与检索片段矛盾，或引用不存在的来源。
原因：大模型在训练中形成了“惯性”，或者 prompt 没有强制约束。
识别：使用忠实度（Faithfulness）指标监控，发现答案与原文不一致。
解决：① 在 prompt 中明确要求“只基于以下文档回答，不要添加外部知识”；② 输出时挂载可点击的来源引用，点击跳转到原始 PDF 行/页。

✅ 最佳实践
  • 精准溯源：每个结论都附带文档 ID、页码、甚至高亮坐标区域。
  • 引用校验：用 RAGAS 等框架自动检查引用页码与原始数据是否匹配。

评估与正确率提升

从 60% 到 85% 的跨越，靠的是系统化诊断和针对性优化。

⚠️ 坑：只看“满意度”或“准确率”单一指标

现象：用户点“不满意”，但不知道是没搜到还是模型胡扯。
原因：指标过于笼统，无法定位瓶颈。
识别：满意度低但检索召回率高，说明生成阶段有问题；反之是检索问题。
解决：拆解指标——Context Recall（答案是否在检索片段中）和 Faithfulness（答案是否忠实于原文）。

✅ 最佳实践
  • NLP 动态切分：不再暴力截断，用句子边界切分 + 重叠窗口（准确率 +15%）。
  • 查询扩写 + 语义校验：用户短问题扩写后，用向量相似度（阈值 0.8）校验，防止扩写模型带偏。
  • 混合排序统一模型：LambdaMART 代替人工加权。
  • 引入 RAGAS 框架：自动化跑分，监控检索命中率、忠实度、引用精确度。
