大模型最让人头疼的
3 个老毛病

ChatGPT 类大语言模型很强，但有 3 个根本性缺陷：

❌ 知识截止——训练数据停在某一天，之后发生的事它不知道
❌ 幻觉——为了让回答完整，它会编造不存在的论文/判例/数据
❌ 私域盲区——没有企业内部数据，不知道产品手册/合同/流程

🔑 解决方案

不是让模型"变聪明"，而是给模型"配个图书检索员"——在它开口之前，先递给它真实资料。这套方法就是RAG（检索增强生成）。

神谕王国与
博学老人

👴 故事开场
远古王国有一位博学老人，读遍所有典籍、答对所有难题。但他记忆停在 3 年前，会编造不存在的法令，对密档一无所知。
⚔️ 危机
邻国入侵，国王急问粮草/兵力/密约。博学老人卡壳："呃……根据历史，三个月吧……"
🧙‍♀️ 救星
索引女巫登场——她从不读内容，但记录所有档案的"索引"。3 秒内找到最相关的 5 段资料。
💡 点题
博学老人 + 索引女巫 = LLM + 检索系统，这就是 RAG。

什么是 RAG
一句话定义

📖 定义
RAG = Retrieval-Augmented Generation（检索增强生成）
🔄 工作流（5 步）
  • 用户提问：把问题输入系统
  • 检索：从知识库里找最相关的 3-5 段文档
  • 组装：把段落+问题拼成新的 Prompt
  • 生成：LLM 基于真实段落组织答案
  • 引用：附上"参考文档：XX 第 X 段"让用户核实
💡 核心洞察
不是让模型"变得无所不知"，而是让模型"问问题时能找到对的资料"。

6 步技术管道
离线 + 在线

📥 离线索引（一次性）
  • 数据采集：读取 PDF / Word / 网页 / 数据库
  • 文档切片：长文档切 200-800 字小段（保留 10-20% 重叠）
  • 向量化：每段转 1536 维浮点数（Embedding）
  • 入库：存到向量数据库（Pinecone / Chroma / Milvus）
🔍 在线检索（每次提问）
  • 查询向量化：用户问题 → 同一维度向量
  • Top-K 检索：找最相似的 K 段（一般 K=3-5）
  • 重排序：用 Cross-Encoder 精排 Top-K
  • Prompt 组装：段落 + 问题 → 喂给 LLM

优缺点
没有银弹

✅ 6 大优势
  • 准确率高：基于真实文档，不易编造
  • 知识可更新：文档更新立刻能用
  • 可解释性强：附引用来源，用户可核实
  • 私域支持：可用企业数据，不需重训
  • 成本低：比 Fine-tuning 便宜得多
  • 安全可控：数据不外发，本地推理
❌ 5 大局限
  • 依赖检索质量：检索错了一切白搭
  • 上下文有限：段落太多 Token 爆掉
  • 切片难：太大检索不准，太小丢语义
  • 多模态弱：图片/表格/公式处理差
  • 冷启动慢：新文档需建索引

5 行代码
跑通 RAG 入门

🐍 LangChain + Chroma
# pip install langchain chromadb openai from langchain.document_loaders import DirectoryLoader from langchain.text_splitter import RecursiveCharacterTextSplitter from langchain.embeddings import OpenAIEmbeddings from langchain.vectorstores import Chroma from langchain.chains import RetrievalQA from langchain.llms import OpenAI # 1. 加载文档 docs = DirectoryLoader('./docs', glob="**/*.pdf").load() # 2. 切片 texts = RecursiveCharacterTextSplitter( chunk_size=500, chunk_overlap=50 ).split_documents(docs) # 3. 向量化 + 入库 db = Chroma.from_documents(texts, OpenAIEmbeddings()) # 4. 检索 + 生成 qa = RetrievalQA.from_chain_type( llm=OpenAI(), retriever=db.as_retriever(k=3) ) # 5. 提问 print(qa.run("年假怎么请？"))
🚀 一句话
从 0 到能跑，只要 5 行核心代码。生产级需加缓存、Rerank、监控。
