CHAPTER 01

🎯 6 大应用场景

RAG 不是"万能胶水"，但在以下 6 个场景里几乎无可替代——只要你的痛点是"数据已经在企业里、但员工找不到"，RAG 就能发挥价值。

📚 企业知识库问答
让员工用自然语言查询公司制度、产品手册、流程规范。覆盖 HR、IT、行政、研发等所有部门。
"我们公司的年假怎么请？""XX 产品的保修期多久？""报销标准是什么？"
💬 智能客服
电商、银行、运营商用 RAG 驱动客服 Agent，基于产品手册、FAQ、历史工单回答用户问题。
"我的订单 12345 什么时候发货？" → RAG 先查订单系统，再查物流 API，最后组织自然语言回答。
⚖️ 法律/合规咨询
律师事务所用 RAG 检索判例法、合同模板、法规条文，辅助律师起草文件、审核合同。
某法律科技公司通过 RAG 把合同审核时间从 4 小时缩短到 20 分钟。
🏥 医疗辅助决策
医生问"这个症状符合哪些鉴别诊断"，RAG 从医学文献库、临床指南、相似病例中检索，给出参考。
注意：医疗 RAG 必须保留医生最终决策权，模型只是"第二意见"。
💻 代码库问答
让开发者用自然语言查询代码库："用户登录的逻辑在哪里？""订单超时的处理在哪？"
GitHub Copilot、Cursor 的 codebase indexing 都是这个思路。
🎓 教育/研究辅助
学生用 RAG 查询教材、论文、笔记；研究员用 RAG 做跨领域文献综述。
NotebookLM 是这个场景的代表产品——把 10 篇 PDF 丢进去，问它综合分析。
CHAPTER 02

⚖️ 优缺点：没有银弹

RAG 不是银弹。在动手前，先看清它擅长什么、不擅长什么——避开 90% 的项目踩坑。

✅ 6 大优势
  • 解决幻觉——答案基于真实文档，模型不再是"凭空编造"
  • 知识可更新——改文档就改答案，不用重新训练模型
  • 答案可追溯——每条答案都能附引用来源，可信度极高
  • 部署快——基于现有文档，2-4 周就能上线 MVP
  • 成本低——比 Fine-tuning 便宜 5-10 倍，效果相当
  • 数据安全——文档可以留在企业内部私域，不上传外部
❌ 5 大局限
  • 依赖文档质量——垃圾进、垃圾出（Garbage In, Garbage Out）
  • 不能跨文档"推理"——单文档检索强，跨文档综合弱
  • 不能处理"没文档的问题"——超出知识库范围的提问无解
  • 有检索延迟——单次查询 1-3 秒，比 Fine-tuning 慢
  • Token 消耗高——每条问题都要把检索结果塞进 Prompt
CHAPTER 03

🏗️ 技术架构：6 步管道

RAG 系统在技术实现上分为"离线索引"和"在线检索"两条线。理解这两条线，就理解了 RAG 的全部工程化逻辑。

📦 离线索引（一次性或定期）

  • ① 文档加载：从 PDF/Word/Confluence/Notion 加载所有企业文档
  • ② 文档切片：按 200-800 字 + 10-20% 重叠切分（法律按条款、技术按段落）
  • ③ 向量化：用 Embedding 模型把每段文本转成 768-1536 维向量
  • ④ 入库：向量 + 原文 + 元数据（作者、时间、部门）一起存到向量数据库

⚡ 在线检索（每次用户提问）

  • ⑤ 查询改写 + 检索：用户问题改写后向量化，从向量库找 Top-K 相似文档（K 通常 3-10）
  • ⑥ 重排 + 生成：用 Rerank 模型精排 Top-K，把段落+问题塞进 Prompt 让 LLM 生成答案 + 引用

CHAPTER 04

🛠️ 实现方式：3 档技术栈

不同场景选不同档位。1 天能上手的 PoC、2 周能上线的生产版、强合规领域需要的自研框架——按你的需求选。

🟢 入门档：LangChain + Chroma（1 天能上手）

适合快速验证想法、学习原理、PoC 阶段。

# 安装 # pip install langchain chromadb openai from langchain.document_loaders import DirectoryLoader from langchain.text_splitter import RecursiveCharacterTextSplitter from langchain.embeddings import OpenAIEmbeddings from langchain.vectorstores import Chroma from langchain.chains import RetrievalQA from langchain.llms import OpenAI # 1. 加载文档 loader = DirectoryLoader('./docs', glob="**/*.pdf") documents = loader.load() # 2. 切片 text_splitter = RecursiveCharacterTextSplitter( chunk_size=500, # 每段 500 字 chunk_overlap=50 # 段间重叠 50 字 ) texts = text_splitter.split_documents(documents) # 3. 向量化 + 入库 embeddings = OpenAIEmbeddings() vectorstore = Chroma.from_documents(texts, embeddings) # 4. 检索 + 生成 qa = RetrievalQA.from_chain_type( llm=OpenAI(), chain_type="stuff", retriever=vectorstore.as_retriever(search_kwargs={"k": 3}) ) # 5. 提问 result = qa.run("我们公司的年假怎么请？") print(result)
🟡 进档：LlamaIndex + Pinecone（生产可用）

适用场景：文档量大（>10万份）、需要长期维护、多用户并发。

  • 用 LlamaIndex 做更精细的索引（树状、关键词表）
  • 用 Pinecone 做生产级向量库（自动扩展、99.9% SLA）
  • 加 Metadata Filtering（按部门/时间过滤）
  • 加 Rerank（用 Cohere Rerank 精排 Top-K）
  • 加缓存（高频问题直接返回）
🔴 专业档：自研 RAG 框架（金融/医疗）

适用场景：金融、医疗、法律等强合规领域——容错率低、可解释性要求高。

  • 混合检索（BM25 关键词 + 向量语义）
  • 多路召回 + 投票机制
  • 幻觉检测（用 LLM 检查答案是否有事实依据）
  • 审计日志（每个回答记录来源、置信度、用户）
  • 权限控制（不同角色看到不同文档）

🟢 案例 1 Glean Technologies · 硅谷明星创业公司，融资 2.6 亿美元，估值 45 亿

Glean——企业搜索领域的独角兽

📍背景

现代企业的痛点：知识散落在 50+ 个工具里——Slack、Notion、Confluence、Google Drive、GitHub、Jira、Salesforce、Figma……员工每天花 2-3 小时找资料，找不到了就问同事。

🔥痛点

传统企业搜索（Elasticsearch）只能搜"关键词匹配"，但员工的问题往往是模糊的："上季度客户反馈最多的功能问题是什么？"——关键词搜不到，因为没人写过这句话。

💡方案
  • 多源接入：接入企业所有办公工具的 API，每天抓取最新内容
  • 权限同步：用权限系统（Google Workspace SSO）确保员工只能搜到有权限看的文档
  • 混合检索：用 Embedding + BM25 混合检索，支持自然语言模糊查询
  • 来源追溯：LLM 生成答案时附上"来源链接"，点击即可跳转到原始文档
🔧技术细节
  • 向量库自研 Glean Index，支持每天增量更新 1 亿+ 文档
  • 检索时用"个人化排序"——根据员工的角色、历史行为优化结果
  • LLM 用 GPT-4 + Claude 双模型，交叉验证答案

⚖️ 案例 2 Harvey AI · 总部在旧金山，YC 投资的法律 AI 独角兽，估值 15 亿美元

Harvey——颠覆律师事务所的 AI 助手

📍背景

法律行业是 RAG 应用的"完美场景"：① 知识是高度结构化的（判例、法规、合同模板）；② 准确性要求极高（错一个判例就可能输官司）；③ 律师平均每小时收费 500-1000 美元，效率提升直接转化为收入。

🔥痛点

传统律所的工作流：初级律师花 5-8 小时检索相关判例、整理判要、写 Memo 草稿，高级律师审阅。这个过程 70% 时间是"找资料"，只有 30% 是"思考"。

💡方案
  • 判例库接入：LexisNexis、Westlaw 等判例库 + 各州法规 + 律所内部历史案件库
  • 精准检索：律师问"加州雇佣合同的违约金上限是多少"，Harvey 自动检索相关判例 + 法规
  • 引用生成：生成答案时附引用："根据 Smith v. Jones (2019) 第 45 段……"
  • 可追溯验证：高级律师可一键追溯到原始判例 PDF 验证
🔧技术细节
  • 法律领域专门微调的 Embedding 模型（基于 LegalBERT）
  • 检索时用 "Rerank + Citation Graph"——不仅看语义相似度，还看判例之间的引用关系
  • LLM 用 GPT-4 Turbo，温度设为 0（法律场景不容创意）

🏥 案例 5 国内三甲医院（隐私原因不便披露）· 医疗

某三甲医院智能问答——医疗 RAG 的本土实践

📍背景

三甲医院的特点：① 内部知识体系庞大——HIS、LIS、PACS 系统 + 10 年电子病历 + 最新临床指南 + 院内制度；② 医生查资料需求高频——"这个患者类似病例 5 年生存率是多少""最新指南对这个病怎么推荐""医院的报销政策是什么"；③ 现有系统（百度、Google）搜不到内部数据，院外 AI 工具又涉及数据合规。

🔥痛点

CHAPTER 06

🎯 能力边界：什么能做、什么不能做

RAG 不是银弹。认清它的能力边界——能做什么、不能做什么、被什么场景误用——是项目选型的第一步。

✅ RAG 擅长的场景
  • 企业知识库问答（制度、流程、产品手册）
  • 技术文档问答（API 文档、代码库）
  • 客户服务（基于产品手册回答用户问题）
  • 法律/医疗研究（基于判例库、文献库）
  • 跨文档综合分析（多文档摘要、对比）
  • 需要可解释性的场景（必须给出来源）
❌ RAG 不擅长的场景
  • 创意写作（诗歌、营销文案、剧情创作）
  • 复杂逻辑推理（数学证明、代码调试）
  • 实时数据查询（股票价格、天气、订单状态——需 API）
  • 跨模态理解（图像、视频、音频）
  • 超出知识库范围的问题（需要世界知识）
⚠️ 经常被误用的场景
  • "想用 RAG 替代搜索"——RAG 适合回答具体问题，不适合"给我看看 X 主题的所有资料"
  • "想用 RAG 做实时决策"——RAG 检索有延迟，关键决策还是用 Agent + 实时 API
  • "文档库里没有的答案"——RAG 不能凭空创造知识，必须基于已有文档
  • "跨语言检索"——中英混杂的文档库需要专门的多语言 Embedding 模型
CHAPTER 07

❓ 5 个常见问题（FAQ）

做 RAG 项目前，这 5 个问题几乎一定会被问到。

Q1: RAG 和 Fine-tuning 哪个好？
不是二选一，是组合拳。RAG 提供事实基础，Fine-tuning 提供语气风格。99% 的企业先做 RAG，效果不够再加微调。
Q2: RAG 的成本主要在哪里？
三块——Embedding 成本（一次性入库 + 增量）、向量数据库存储费、LLM 推理成本。前两者是固定成本，后者是变动成本。
Q3: 用开源 Embedding 还是 OpenAI 的？
中文场景推荐：BGE（智源）、M3E（智源）、BCEmbedding。英文场景：OpenAI text-embedding-3-small。开源更省钱但效果略差。
Q4: 切片大小怎么选？
经验值 200-800 字 + 10-20% 重叠。法律文档按条款切，技术文档按段落切，代码按函数切。
Q5: RAG 效果怎么评估？
4 个核心指标——检索命中率（Recall@K）、答案准确率、引用准确率、用户满意度。必须用黄金数据集测试。
CHAPTER 08

🎁 5 句话总结

如果只能记住 5 句话，记住这 5 句。

1 RAG 不是让模型"变聪明"，而是给模型"配了图书检索员"。
2 RAG 的核心是"检索质量"——检索错了，再强的 LLM 也救不回来。
3 RAG 是企业 AI 的第一站——先做 RAG，效果不够再考虑微调。
4 RAG 不是银弹——创意写作、实时决策、跨模态不擅长。
5 RAG 成功的关键不是技术，是"文档质量"——垃圾进、垃圾出。
